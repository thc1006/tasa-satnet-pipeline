## 3. Hypatia I/O contract (the actual deliverable)

### 3.1 Inputs Hypatia ns-3 expects

A run takes a `config_ns3.properties` plus an upstream **satellite-network-state directory** generated by `satgenpy`.

**`config_ns3.properties`** (concrete example from `run_loaded_tm_pairing_10_Mbps_for_10s_with_udp`):
```ini
simulation_end_time_ns=10000000000                    # 10 s sim
dynamic_state_update_interval_ns=100000000            # routing table refreshed every 100 ms
satellite_network_dir="..../kuiper_630_isls_plus_grid_ground_stations_top_100_algorithm_free_one_only_over_isls"
satellite_network_routes_dir="..../dynamic_state_100ms_for_200s"
isl_data_rate_megabit_per_s=10.0
gsl_data_rate_megabit_per_s=10.0
isl_max_queue_size_pkts=100
gsl_max_queue_size_pkts=100
tcp_socket_type=TcpNewReno
enable_isl_utilization_tracking=true
isl_utilization_tracking_interval_ns=100000000
enable_udp_burst_scheduler=true
udp_burst_schedule_filename="udp_burst_schedule.csv"
```

**`udp_burst_schedule.csv`** (the traffic matrix):
```
id, src_node, dst_node, target_rate_mbps, start_ns, end_ns, ?, ?
0,1214,1250,10.0000000000,0,1000000000000,,
```
100 flows in this run, all 10 Mbps, 1 s each — total 100 simultaneous UDP flows pairing 100 ground stations.

**Satellite network state** (8 files + dynamic state directory, 31 MB total for Kuiper-630):
| File | Rows | Schema |
|---|---|---|
| `description.txt` | 2 | `max_gsl_length_m`, `max_isl_length_m` |
| `tles.txt` | 1 hdr + 3 lines/sat | `<orbits> <sats_per_orbit>` then standard 3-line TLE per satellite, named `Kuiper-630 0`, `Kuiper-630 1`, ... |
| `ground_stations.txt` | 100 | `id, name, lat, lon, alt, ECEF_x, ECEF_y, ECEF_z` |
| `isls.txt` | 2,312 | `<sat_a> <sat_b>` pairs (the +Grid topology) |
| `gsl_interfaces_info.txt` | 1,256 (= 1156 sats + 100 GS) | `node_id, num_interfaces, bandwidth_factor` |
| `dynamic_state_100ms_for_200s/fstate_<ns>.txt` | 125,500 each × 2,000 snapshots | `src, dst, next_hop, my_iface, next_iface` (forwarding state) |
| `dynamic_state_100ms_for_200s/gsl_if_bandwidth_<ns>.txt` | varies × 2,000 snapshots | `node_id, iface_id, bandwidth_factor` (GS visibility over time) |

### 3.2 Outputs Hypatia ns-3 produces

| File | Rows | Schema | Use |
|---|---|---|---|
| `logs_ns3/udp_bursts_outgoing.csv` | 100 (one per flow) | `id, src, dst, target_mbps, start_ns, end_ns, runtime_s, achieved_mbps, packets_sent, bytes_target, bytes_sent` | **Per-flow KPI** — replaces TASA SatNet's 8.91 ms / 40 Mbps physics formula |
| `logs_ns3/udp_bursts_incoming.csv` | 100 | (mirror) | Verifies sender ≈ receiver (loss check) |
| `logs_ns3/isl_utilization.csv` | **25,440** | `src, dst, t_start_ns, t_end_ns, util_fraction` | Per-ISL link load over time — completely new dimension TASA SatNet doesn't have |
| `logs_ns3/timing_results.txt` | 14 lines | breakdown of setup vs sim time | Performance debug |
| `logs_ns3/finished.txt` | 1 word | `Yes` or `No` | Sentinel; check before parsing |
| `logs_ns3/console.txt` | ~kB | Full stdout log | Debug |

**Concrete example of one row in `udp_bursts_outgoing.csv`** from the Kuiper-630 10 Mbps × 10 s run:
```
0,1214,1250,10.000000,0,1000000000000,10.000800,9.814118,8334,12501000,12267648
```
Decoded: flow 0 from node 1214 to node 1250, target 10.0 Mbps, achieved **9.814 Mbps** (1.86% loss vs target), 8,334 packets sent, 12.27 MB delivered out of 12.50 MB intended. **This is what packet-level KPI looks like.**

### 3.3 Key insight for v2 of TASA SatNet

```
Current TASA SatNet pipeline:
  OASIS log ─▶ windows.json ─▶ scenario.json ─▶ metrics.csv (physics formula, 8.91 ms / 40 Mbps for ALL sessions)

v2 with Hypatia:
  OASIS log ─▶ windows.json ─▶ scenario.json ─┬──▶ satgenpy adapter ─▶ {tles.txt, isls.txt, ground_stations.txt, dynamic_state/*}
                                              └──▶ traffic adapter   ─▶ udp_burst_schedule.csv
                                                          │
                                                          ▼
                                                   ns-3 + Hypatia
                                                          │
                                                          ▼  packet-level KPI
                                                   metrics.csv (real per-flow achieved Mbps,
                                                                real per-ISL utilization)
```

The bridge is **two adapter modules**:
- `scripts/adapters/to_satgenpy.py` — `windows.json` + `taiwan_ground_stations.json` + TLE → satgenpy state directory
- `scripts/adapters/from_hypatia.py` — `udp_bursts_outgoing.csv` + `isl_utilization.csv` → TASA-flavoured `metrics.csv` schema

Both are pure Python file format conversion. No deep coupling.

## 4. Cost numbers from the temp_data evidence

### 4.1 Run wall-clock (Kuiper-630, 1156 nodes, 100 UDP flows, 10 s sim)

From `timing_results.txt` of the smallest run:
```
[    0.0 -     1.6] (1.6 s) :: Create initial single forwarding state
[    1.6 -     4.3] (2.7 s) :: Setup routing arbiter on each node
[    4.4 -     4.4] (0.0 s) :: Setup UDP burst applications
[    4.4 -   111.8] (107.4 s) :: Run simulation
Total wall: 111.9 s
```

**107 s wall to simulate 10 s of network** = **~11× slower than real time** for 1156-node Kuiper case. Linear extrapolation: 200 s sim → ~37 min wall; 1 hour sim → ~11 h wall; 24 h sim → ~11 days wall. **Hypatia is not designed for long simulation horizons.**

### 4.2 Disk per run

- **Input state** for one Kuiper-630 config @ 100ms / 200s: **31 MB** (mostly `fstate_*.txt` × 2000)
- **Output for one 10 s run**: ~few MB (100-row UDP CSVs + 25k-row ISL utilization)
- **Full paper data**: 478 MB compressed, ~5 GB extracted

### 4.3 Disk for TASA-scale (extrapolated)

The Kuiper case is 1156 nodes × 2000 snapshots = 2.3 M state entries.

For TASA scale (say 10 satellites × 6 GS = 16 nodes, 1 hour sim @ 1 s state granularity):
- 16 × 16 = 256 routing entries per snapshot × 3600 snapshots ≈ 1 M entries
- **~10–50 MB** of state per scenario — same order of magnitude

For the smallest TASA case (1 sat × 6 GS, 200 s @ 100ms): **< 1 MB**. Truly trivial.

### 4.4 RAM

Not directly observable from temp_data. Community reports (research agent confirmed via issues #9, #20):
- Kuiper-630, 200 s sim: 3–6 GB peak
- For TASA scale (~16 nodes), expect **< 500 MB** — fits comfortably even on this 7 GB-available machine.

### 4.5 Build cost (skipped)

Issue #39 confirms ns-3.31 (Hypatia's pinned version) does NOT build on Ubuntu 24.04 + GCC 13/14. Three workarounds exist:
- Build inside `ubuntu:20.04` Docker container (~30–60 min one-time)
- Patch ns-3 to ns-3.45 (one of the open forks, e.g. issue #39 references)
- Wait for upstream fix (last commit 2024-05-15, bus-factor 1, won't happen)

## 6. Implications for v2 of TASA SatNet

### 6.1 v2 is feasible. Here's the revised cost estimate.

| Phase | Effort | Risk |
|---|---|---|
| **6.1.A** Write `to_satgenpy.py` adapter | 1–2 days | Low — pure file format conversion, schema is documented above |
| **6.1.B** Run satgenpy on TASA TLE + 6 stations to produce `dynamic_state_*` | 1 day | Low — satgenpy is pure Python, install only needs cartopy (libproj-dev) |
| **6.1.C** Build ns-3.31 + Hypatia in `ubuntu:20.04` Docker | 0.5–1 day | Medium — known to work; wrap in Dockerfile so it's a one-time pain |
| **6.1.D** Write `from_hypatia.py` parser | 0.5 day | Low |
| **6.1.E** Replace `metrics.py` physics formulas with `from_hypatia.py` for runs where Hypatia output exists | 1 day | Low |
| **6.1.F** Add `--use-hypatia` flag, defaulting OFF | 0.5 day | Low — preserves option value, lets you A/B compare |
| **6.1.G** Document v2 in README, add example | 0.5 day | Low |
| **Total** | **~5–7 days of focused work** | Mostly low |

This is **half** the original 3-month estimate, because:
1. No need to fork/patch ns-3 — `ubuntu:20.04` container sidesteps Ubuntu 24.04 issues
2. TASA scale is tiny vs. paper scale → runtime is seconds not minutes
3. Pre-existing temp_data lets v2 contract testing happen offline without ever running ns-3

### 6.2 What v2 cannot do

- **Long-horizon simulations** — Hypatia is 11× slower than real time at 1156-node scale. For TASA's typical 24h windows, even at small node counts you're looking at hours of wall time. Acceptable for batch studies, fatal for any "interactive" use case.
- **Mega-constellation simulation** — Above ~2000 nodes / 100 s, RAM and runtime explode. Not a TASA need anyway.
- **Real-time integration with live OASIS log streams** — Hypatia is precompute-then-simulate. Streaming would need a different architecture (xeoverse / StarryNet were designed for this).

### 6.3 Stage 1 + Stage 2 plan, refined

The plan from the previous turn ("Stage 1 = generic LEO backbone, Stage 2 = TASA case study") still holds. **What changes**: the v2 ETA shrinks from "Q3 2026" to "next month if you commit a week of focused work". This is good news.

What I'd do differently knowing this:
- **Drop Hypatia integration as the v2 milestone** — it's a 5-7 day adapter project, not a quarter-long architecture initiative.
- **Keep the v2/v3/etc. roadmap** but compress: v2 is the Hypatia adapters (5–7 days), v3 is the proper stuff — DRL scheduler, MILP baseline, OMM ingest, Argo Workflows.
- **Stage 2 (TASA case study) becomes immediately actionable** — once v2 ships, you can publish a "Taiwan B5G LEO under realistic packet load" demo with < 2 weeks total work.

### 6.4 Should I still try xeoverse / StarryNet?

Recommendation: **yes, but in its own time-box, after v2 with Hypatia ships**. Reasoning:
- Hypatia is the conservative path (most established, most cited, batched precompute model fits TASA's batch nature)
- xeoverse / StarryNet are the optimistic path (newer, faster, container-native, but bus-factor 1 each, no Taiwan precedent)
- Doing Hypatia first gives you a publishable v2 with low risk; doing xeoverse first risks 4 h of "couldn't make it run" with nothing to ship

## 7. Recommendations to the project

1. **Don't repeat my time box.** Start any future v2 work by extracting `hypatia_paper_temp_data.tar.gz` and reading §3 above as the contract spec. The release tarball is the cheapest way to learn Hypatia.
2. **Pin a Dockerfile now.** Even before any v2 code, commit a `docker/hypatia-base.Dockerfile` based on `ubuntu:20.04` that builds Hypatia/ns-3 once. This is the load-bearing risk — derisk it early.
3. **Skip cartopy** in the satgenpy install if you don't need plotting — saves ~10 min of cartopy compilation against libproj-dev.
4. **Adopt Hypatia's filename conventions** in the adapter outputs (`fstate_<ns>.txt`, `gsl_if_bandwidth_<ns>.txt`, `udp_burst_schedule.csv`) — don't invent your own. The whole ecosystem assumes these names.
5. **Don't try to wrap Hypatia inside the K8s job** for the first iteration. Run it on bare hardware or a single VM. K8s makes ns-3 builds painful (no privileged containers etc.).
